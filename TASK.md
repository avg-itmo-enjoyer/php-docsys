# Тестовое задание
## Задача
Необходимо реализовать backend-сервис по работе с документами. 
Документы создаются, переводятся по статусам. 
По изменениям статусов ведётся журналирование.

Дополнительно требуется разработать утилиту для массового создания документов.
Также требуется реализовать batching-обработку документов

## Функциональные требования
### Документ
**Поля документа:**
* Внутренний id
* Уникальный номер, генерируемый при создании
* Автор
* Название
* Статус
* Дата создания
* Дата последнего обновления

**Возможные статусы:**
* DRAFT
* SUBMITTED
* APPROVED

### Журнал
При каждом переводе статуса в журнал сохраняется запись.  

**Структура записи:**
* Инициатор
* Тип инициированного действия
* Временна́я метка
* Комментарий (должен быть опциональным)

**Действия:**
* SUBMIT
* APPROVE.

Для всех запросов на создание/перевод статуса передаётся инициатор

### Реестр утверждений
При успешном утверждении документа создаётся запись в реестре утверждений
(отдельная таблица/хранилище в рамках проекта).
Если запись в реестр создать не удалось, утверждение документа должно быть
отменено.

## Описание API:
1. **Создание документа**
2. **Получение документов**
3. **Отправка на согласование**
4. **Утверждение документа**
5. **Поиск по документам**
6. **Проверка конкурентного утверждения**

## Дополнительные модули и утилиты
### Утилита для генерации
Требуется создать отдельный модуль/утилиту, которая читает из файла параметров число N и
создаёт N документов через API сервиса.

### Фоновые процессы
В сервисе должны работать два фоновых процесса (в отдельных потоках/задачах).
Размер batch'а должен задаваться параметром batchSize (в конфиге/файле
параметров) и использоваться в обработке.
1. **SUBMIT-worker**  
   Регулярно проверяет БД и отправляет документы со статусом DRAFT на
   согласование пачками по batchSize через пакетное API.
2. **APPROVE-worker**  
   Регулярно проверяет БД и отправляет документы со статусом SUBMITTED на
   утверждение пачками по batchSize через пакетное API.

Частичные ошибки не должны останавливать обработку.

## Формат логов
По логам должно быть понятно:
* Сколько документов задано к созданию (N) и прогресс создания;
* Время выполнения шагов: создание, каждая отправка пачки на согласование,
каждая отправка пачки на утверждение;
* Ход фоновой обработки (сколько документов обработано/осталось).

## Ошибки и базовая валидация
* Единый формат ошибок: код + сообщение.
* “не найдено” для отсутствующего документа.
* “конфликт/недопустимая операция” для недопустимого перехода статуса.
* Проверка входных данных (пустые значения и т.п.).

## Тесты
Минимальный набор тестов должен содержать:
* Happy-path по одному документу
* Пакетный submit
* Пакетный approve с частичными результатами
* Откат approve при ошибке записи в реестр утверждений.
